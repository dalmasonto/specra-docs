## How It Works

The API documentation system uses **auto-detection** by default. Simply reference your spec file and the system automatically determines the format:

```mdx
<ApiReference spec="/api-specs/my-api.json" />
```

The parser will detect whether your file is:
- ‚úÖ Specra format (native)
- ‚úÖ OpenAPI 3.x or Swagger 2.0
- ‚úÖ Postman Collection v2.0 or v2.1

## Choosing a Format

### Use Specra Format When:

- üìù Writing docs from scratch
- üéØ Need a simple, minimal format
- üöÄ Want to get started quickly
- üìñ Documenting < 20 endpoints

**Example:**
```json
{
  "title": "My API",
  "baseUrl": "https://api.example.com",
  "endpoints": [...]
}
```

### Use OpenAPI When:

- üè¢ Following enterprise standards
- üîß Generating specs from code
- üîÑ Sharing specs across tools
- üìä Need detailed schema validation
- üåê Working with API gateways

**Example:**

```json
  "info": {...},
  "paths": {...}
}
```

- üß™ Your team uses Postman for testing
- üíæ Have existing Postman collections
- üîÅ Want docs in sync with tests
- üì∏ Need saved response examples
- ‚ö° Want quick export workflow

**Example:**

## Workflow Examples

### Workflow 1: Start Simple (Specra)

Perfect for new projects or small APIs:

1. **Create a Specra JSON file** in `/public/api-specs/`
2. **Add your endpoints** with examples
3. **Reference in MDX**:
   ```mdx
   <ApiReference spec="/api-specs/my-api.json" />
   ```
4. **Done!** Interactive docs are live

---

### Workflow 2: Use Existing OpenAPI (OpenAPI)

Great if you're already using OpenAPI:

1. **Generate OpenAPI spec** from your code (e.g., FastAPI, Swagger, etc.)
2. **Place in `/public/api-specs/`**
3. **Reference in MDX**:
   ```mdx
   <ApiReference spec="/api-specs/openapi.json" />
   ```
4. **Keep in sync** by regenerating when code changes

---

### Workflow 3: Export from Postman (Postman)

Ideal for teams using Postman:

1. **Develop and test** your API in Postman
2. **Add response examples** in Postman
3. **Export collection** (v2.1 format)
4. **Place in `/public/api-specs/`**
5. **Reference in MDX**:
   ```mdx
   <ApiReference spec="/api-specs/collection.json" />
   ```
6. **Re-export** when you update your collection

## Mixed Approach

You can use multiple formats in the same project:

```mdx
<!-- Auto-generated OpenAPI for main API -->
<ApiReference spec="/api-specs/main-api-openapi.json" />

<!-- Postman collection for external API -->
<ApiReference spec="/api-specs/third-party-postman.json" />

<!-- Specra for internal tools -->
<ApiReference spec="/api-specs/internal-tools-specra.json" />
```

## Advanced Usage

### Explicit Parser Selection

While auto-detection works well, you can explicitly specify the parser:

```mdx
<ApiReference spec="/api-specs/my-api.json" parser="openapi" />
<ApiReference spec="/api-specs/my-api.json" parser="postman" />
<ApiReference spec="/api-specs/my-api.json" parser="specra" />
```

### Disable Playground

Turn off the interactive playground if you don't need it:

```mdx
<ApiReference spec="/api-specs/my-api.json" showPlayground={false} />
```

### Multiple Versions

Document multiple API versions:

```mdx
## Version 2.0 (Current)

<ApiReference spec="/api-specs/api-v2.json" />

## Version 1.0 (Deprecated)

<Callout type="warning">
This version will be sunset on June 1, 2024
</Callout>

<ApiReference spec="/api-specs/api-v1.json" showPlayground={false} />
```

## Common Features

Regardless of which format you choose, all specs support:

‚úÖ **Path parameters** - Dynamic URL segments (`:id`, `{id}`)

‚úÖ **Query parameters** - URL query strings

‚úÖ **Request headers** - Custom headers

‚úÖ **Request bodies** - JSON payloads

‚úÖ **Response examples** - Sample responses

‚úÖ **Multiple response codes** - 200, 404, 500, etc.

‚úÖ **Authentication** - Bearer, API Key, Basic

‚úÖ **Interactive playground** - Test APIs live
